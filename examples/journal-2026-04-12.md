seems to be: deploy in shadow mode for a week, compare counter values
against the legacy implementation, then flip the enforcement bit. The

deploy. Decided to delegate to Cem, who's been thinking about this

Pair-debugged a flaky integration test with Esra. The flake was a
classic test-isolation bug — one test was leaving rows in the database
that another assumed were absent. Took me embarrassingly long to spot.
Lesson: when a test is flaky, check the database state before assuming
it's a race condition.

Started reading the qmd README again, because we keep coming back to
the question of "should we just use qmd" vs "should we build our own
search". The answer remains: we want a single Go binary, no Node
runtime, and an in-process Go API. So we build our own and credit qmd
for the algorithms. (See `technical-doc.md` for the architectural notes.)
