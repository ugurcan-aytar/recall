# Incident Report — 2026-03-22 — Rate Limiter Exhaustion

Severity: SEV-2.
Duration: 2026-03-22 14:08 → 14:51 UTC (43 minutes).
Customer impact: ~12% of API requests rejected with HTTP 429.

## Summary

A spike in legitimate traffic from a newly onboarded customer pushed
our gateway-side rate limiter past its capacity. The limiter started
rejecting unrelated traffic because the redis instance backing it ran
out of memory and began evicting counters.

## Timeline

- 13:55 — New customer (TenantID `acme-co`) ramps up usage 5× over
  their previous baseline.
- 14:08 — `RateLimiterRedisMemoryHigh` alert fires. No on-call action;
  the alert was set to "warn" only.
- 14:14 — Customer support reports unrelated tenants seeing 429s.
- 14:18 — On-call (Deniz) acknowledges, opens incident channel.
- 14:25 — We identify the new tenant as the cause via the
  `tenant_request_rate` Prometheus query.
- 14:32 — We bump the redis instance from `cache.r6g.large` to
  `cache.r6g.xlarge`. Counters are still being evicted.
- 14:41 — We temporarily lift the global limiter ceiling, accepting
  the risk of overload, while we investigate.
- 14:47 — Counters stabilise after the redis instance scales up.
- 14:51 — Error rate returns to baseline. Incident closed.

## Root cause

The redis maxmemory-policy was `allkeys-lru`, which means under memory
pressure it evicts ANY key — including the rate limiter counters. When
counters get evicted, requests look like "first request for this token
bucket" and the limiter resets. The compounding effect is that the
limiter under-counts the busiest tenants, then over-corrects when the
counter eventually re-establishes.

## Action items (status as of retro)

1. ✅ Switch redis to `volatile-lru` so the limiter counters (which
   have TTLs) are evicted but session/cache keys aren't preferred.
2. ✅ Promote `RateLimiterRedisMemoryHigh` from warn to page.
3. ⏳ Add per-tenant alerting: `TenantRequestRateAnomaly` (open: Cem).
4. ⏳ Add a queue-depth metric for the limiter (open: Cem).
5. ✅ Update the runbook (`runbook-payments.md`).

## Lessons

- We didn't have alerting for the queue depth. A high-cardinality
  tenant should have been visible before users felt anything.
- The redis eviction policy was set in 2024 and never revisited.
  Default policies need an annual review.
- Customer onboarding does not currently flag "this tenant will 5×
  our load" — sales should loop us in for tenants with large
  projected volume.
